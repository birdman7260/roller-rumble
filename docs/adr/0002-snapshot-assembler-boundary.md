# The snapshot assembler is a pure read-only module; the app owns reconciliation and the server owns throttling

The `SnapshotAssembler` module is constructed to be a pure read-only operation that owns no timers, sockets, database writes, or side effects. It takes a `SnapshotContext` containing only the live runtime state that `RollerRumbleApp` knows at assemble time—tunnel state, OS2L diagnostics, photo-booth status, Stripe setup, result presentation, countdown duration lookup, and an injectable clock for testing determinism—and produces the full `AppSnapshot` from SQLite plus that context. By injecting the clock and accepting pre-reconciled queue state, the module avoids coupling to external timing or mutable state, making it testable in isolation against golden snapshots.

`reconcileQueueRaceStatuses` is a database write that modifies queue entry state. It was kept out of the assembler to preserve the module's purity and is now called as a pre-step in `RollerRumbleApp` before invoking `assemble`. This placement reflects the architectural boundary: the app owns the decision to mutate state, and the assembler only reads the result. The same reconciliation function is shared with two other non-snapshot call sites, so it made sense to keep it in the app rather than duplicate it or pollute the assembler.

Per-client snapshot throttling and diffing—specifically the 1000ms racer message coalescing timer that reduces redundant WebSocket writes—stays in `server.ts` because it is connection-scoped and architectural to the transport layer, not part of the snapshot shape or its derivation. The server still owns the decision of when and how to send snapshots to clients, while the assembler owns only the shape of what is sent.

The `SnapshotStreamSurface` type (`"admin" | "projector" | "racer"`) carries only a surface name on the WebSocket connection; there is no racer ID parameter. The racer payload is a single public-safe projection of the full snapshot that strips operator-only state (tunnel diagnostics, OS2L detail, photo-booth status, Stripe configuration, live metrics, result presentation, and ticker messages). One payload serves all racers, matching the prior behavior where the app did not differentiate racer clients.

The `SnapshotAssembler` class is constructed and owned by `RollerRumbleApp`. The per-surface projection is re-exposed as `service.snapshotForSurface(snapshot, surface)` so `server.ts` depends only on the service, not on the snapshot module directly. This keeps the server layer from knowing the snapshot shape and reduces coupling.

A golden-snapshot fixture and an injected clock guard the extraction. The wire payload must remain byte-identical to the prior serialization so existing racer phones, cached assets, and integration tests do not break unexpectedly.
