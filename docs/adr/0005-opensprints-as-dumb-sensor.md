# OpenSprints hardware is driven as a dumb tick-streamer, and located by protocol handshake

The race hardware is an **OpenSprints** unit — an Arduino behind a USB-to-serial chip (a COM port at 115200 8N1), with a reed switch per bike roller; one pulse = one **tick** = one roller revolution. It is open source, so the serial protocol is known rather than reverse-engineered (see `docs/opensprints-protocol.md`). The firmware is not a passive sensor: it runs its own race lifecycle — a fixed ~4-second `CD:` countdown, its own distance/time finish detection (`0F:`…), and cosmetic false-start reporting (`FS:`) — and only streams `R:<t0..t3>,<ms>` progress _after_ it receives a `g` (GO) command.

We decided to **treat the box as a dumb tick source** rather than let it run the race. `ActiveRace` remains the single owner of countdown, finish line, and finalization, exactly as it is for the `SimulatorSensorAdapter`. The USB adapter sends `g` only to make the box start streaming, sets the box's tick-length high enough that the box never declares its own finish, ignores `0F:`/`FS:`, feeds raw tick deltas into the existing `deltaRotations` seam, and sends `s` (STOP) on `endRace()` to silence the box. We chose this because the app's architecture already centralizes race lifecycle in `ActiveRace`, and keeping the USB and simulator adapters interchangeable behind one `SensorAdapter` contract is worth more than reusing the hardware's built-in race features. The rejected alternative — box as race controller, app as a thin shell consuming `CD:`/`0F:` as truth — would couple our race lifecycle to firmware quirks (fixed countdown, baked-in false-start rules) and require a real `ActiveRace` refactor.

Because the protocol is **self-identifying** (`v` → `V:…` / `basic-1`), production port detection is a **probe-and-confirm handshake**: open candidate COM ports, send `v`, and accept only a port that replies with a recognized version. VID/PID is used only to _prioritize_ which ports to probe first, never as a hard filter, so a board revision can't lock us out. This is deliberately not SilverSprint's "grab a port matching `Arduino.*`, else the last port and hope" approach, which can bind to the wrong serial device.

## Consequences

- The `SensorAdapter` contract gains a **lifecycle/countdown channel** (beyond `RotationEvent`) so the app's countdown can mirror the box's real `CD:` messages closed-loop. Countdown ownership becomes a per-adapter property: the simulator keeps owning its own countdown; the USB adapter defers to the box. The app's configurable countdown duration is therefore overridden by the box's fixed 3‑2‑1‑GO on the USB path.
- One tick = one roller revolution, so distance depends on the **roller rollout**, not a bike-wheel circumference. This is a managed setting ("meters per roller revolution"), seeded from a real measurement, modeled to allow a future per-lane override.
- The box reports four positional tick counts regardless of bike count, so **lane mapping** (`sensor index → lane`) is a managed setting, not hardcoded — a flipped mapping is invisible and would crown the wrong winner.
- The box zeroes its tick counters on every `g`, so a **mid-race reconnect cannot cleanly resume**; the honest behavior is to mark the race interrupted rather than silently restart the box.
- The USB sensor joins tunnel/Stripe/OS2L on the **subsystem health** surface, with a 1s reconnect loop and a manual COM-port override as the escape hatch for a live event.
