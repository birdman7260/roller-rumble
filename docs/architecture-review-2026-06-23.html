Problem

Understanding "when does a race end" requires tracing 4 functions across 150 lines. The lifecycle seam is invisible — callers see individual steps, not one module.

Solution

Extract an ActiveRace module whose interface is start / tick(rpm) / finalize. Implementation owns all lane state, metric computation, and result persistence.

server.ts: snapshot.forAdmin() · snapshot.forRacer(id)

· stat computation

· queue reconcile

· racer payload filter

· throttle/diff state

Problem

Snapshot shape knowledge leaks across the seam: server.ts re-filters the snapshot for racer clients and throttles per-client diffs. Changing snapshot shape requires touching two files.

Solution

A deep SnapshotAssembler module exposes forAdmin() and forRacer(id). Throttle state, racer payload filtering, and stat computation are implementation details.

Wins

  • · locality: snapshot shape concentrated in one module
  • · seam leak from server.ts eliminated
  • · leverage: add a new surface without touching server.ts
  • · tests can assert racer vs admin payloads directly

4 · Make result presentation side effects explicit

Worth exploring in-process

apps/desktop/src/backend/services/app.ts · lines 769–815

Before — side effects hidden in "clear"

After — one method, named for what it does 

sequenceDiagram

  participant A as app.ts

  C->>A: dismissResultAndAdvanceQueue()
  Note over A: all four effects happen,
  Note over A: all named in one place

Problem

clearRaceResultPresentation() names only one of four effects. The other three (queue completion, auto-stage, notifications) are invisible at the call site — the interface misrepresents the implementation.

Solution

Rename to dismissResultAndAdvanceQueue() and consolidate the three implicit triggers into one explicit place. The interface then matches the implementation's actual depth.

Wins

  • · interface matches implementation depth
  • · test name becomes accurate ("dismiss triggers auto-stage")
  • · locality: all four effects in one function

After — stream emits events; queries self-subscribe 

flowchart TD
  WS[WebSocket stream]
  EB[snapshotEvents emitter]
  Q1["useTunnelQuery — subscribes to 'tunnel'"]
  Q2["useNotifQuery — subscribes to 'notification'"]
  Q3["useEnvQuery — subscribes to 'env'"]

  WS -->|"emits typed events"| EB
  EB --> Q1
  EB --> Q2
  EB --> Q3

Problem

The stream hook is shallow: its interface is simple, but it contains growing knowledge of which query keys to invalidate for each snapshot field. Adding a new derived query requires modifying the stream.

Solution

The stream emits typed events. Each query hook subscribes to the relevant event. Adding a new query does not touch the stream module — it's a new subscriber at the seam, not a modification of it.

Wins

  • · leverage: stream seam stays stable as queries grow
  • · two subscriber shapes justify the seam
  • · speculative: current query count may not warrant this yet

Top recommendation

Start with the race lifecycle module

Candidate 1 — Concentrate the race lifecycle — is the highest-leverage first move. It's the most-tested, most-changed code path; extracting it into one deep ActiveRace module immediately improves locality for the most active bugs, creates a natural test surface for the full race arc, and unblocks Candidate 3 (god class split) by pulling the biggest, most tangled slice out first.

Candidates 2 and 3 should follow in sequence — each deepening makes the next cut easier.
